## Version Rollback Incident (2:00 - 14:30)

**Derek Liu:** So last Thursday — March 13th — one of our engineers was working on our customer health scorecard project. The one we deployed through the One-Click Deploy beta. He'd been iterating on the frontend and made a generation that broke some styling. Normal stuff. He went to version history to roll back to the previous version.

**Hannah:** Okay.

**Derek Liu:** The rollback appeared to succeed. Forge showed the previous version restored, the preview looked right, the app re-deployed automatically. Everything seemed fine. But 40 minutes later, our on-call engineer gets paged. Three API endpoints are returning 404s. The customer data endpoint, the health score calculation endpoint, and the alert threshold endpoint. All gone.

**Jordan Kim:** Gone as in the route definitions were missing from the code?

**Derek Liu:** Exactly. The rollback restored the frontend components and most of the backend, but it silently dropped three API route files. No error, no warning, no diff showing what changed. The engineer had no way of knowing that the rollback was incomplete. We were down for 40 minutes before anyone noticed, and it took another 20 minutes to figure out what happened and manually restore the endpoints from a git backup.

**Jordan Kim:** That's a serious issue. Do you still have the project ID and the version timestamps? I want to reproduce this in our environment.

**Derek Liu:** Priya has all the details. She did the post-mortem.

**Priya Nair:** I'll send everything over today — project ID, the two version hashes, and our internal incident report. The root cause from our side is clear: the rollback didn't preserve the full file tree. It restored the files that existed in both versions but dropped files that were added between the two versions and were still needed.

**Jordan Kim:** So it treated the rollback as a snapshot restore rather than a diff-based revert. That's a bug in how we handle file additions across versions. I can see how that would happen with our current implementation.

**Hannah:** Jordan, what's your confidence level that we can fix this?

**Jordan Kim:** High. This sounds like a specific edge case in how we serialize the file manifest during rollback. I'll need to dig into it, but architecturally it's a solvable problem. The harder question is whether there are other edge cases like this we haven't found yet.

**Derek Liu:** And that's what worries me. We had a production incident last Thursday where an engineer tried to rollback to a previous version and it silently dropped three API endpoints. We were down for forty minutes before anyone noticed. Version history can't just be a nice-to-have — it needs to be rock solid. If I can't trust rollback, I can't trust Forge for production workloads. We've deployed five apps now. Every one of them could have this same issue.

**Hannah:** Derek, I hear you, and I want to validate that concern. You're right that rollback needs to be bulletproof if we're going to be a production platform. I'm going to escalate this internally as a P0 bug.

## Version Diff Discussion (14:30 - 21:00)

**Hannah:** Priya, you mentioned in the pre-call notes that the version history UI made this harder to catch. Can you walk us through that?

**Priya Nair:** Yeah. So when you open version history in Forge today, you see a list of versions. Each one has a timestamp and a label if you added one. That's it. There's no way to see what changed between version A and version B. No diff view, no file list, nothing.

**Jordan Kim:** You'd want something like a GitHub PR diff?

**Priya Nair:** Exactly. The version history UI shows me a list of timestamps, but I can't see what actually changed between versions. I need a diff view — show me what lines were added, removed, modified. Without that I'm rolling back blind. If our engineer had been able to see a diff before confirming the rollback, he would have noticed that three API files were about to be removed. He would have caught it before it went to production.

**Hannah:** That makes total sense. A diff view would serve as both a diagnostic tool and a safety check.

**Priya Nair:** Right. And beyond rollback, it's useful for general development. When I'm iterating on a project — prompt, generate, review, prompt again — I want to see what each generation actually changed. Sometimes the model modifies files I didn't ask it to touch. A diff view catches that.

**Jordan Kim:** We've had a version diff feature on the backlog for a while. I think this incident elevates the priority. Hannah, we should probably spec this out alongside the rollback fix.

**Hannah:** Agreed. I'll draft a spec this week and share it with Priya for feedback.

**Priya Nair:** Happy to review. I have a lot of opinions about what a good diff UX looks like. I've used GitHub, GitLab, and Bitbucket extensively and they all have different trade-offs.
