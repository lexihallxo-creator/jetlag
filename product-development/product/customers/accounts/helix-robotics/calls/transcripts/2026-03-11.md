## Hardware Review Workflow (2:30 - 12:45)

**Rachel Torres:** Sure. So the biggest thing on my mind right now is collaboration. We've grown to 15 engineers, and we're spread across three offices — Austin, Munich, and Shenzhen. Our hardware design review meetings are the heartbeat of our development process. Every week, we get engineers from all three sites on a call to review the latest prototypes — dashboards, control panels, monitoring interfaces, all built in Forge.

**Hannah:** How are those review sessions going today?

**Rachel Torres:** Honestly? They're chaos. Here's what happens. We get on a Zoom, someone shares their screen, they walk through the prototype, people give feedback verbally, then they try to make edits while everyone watches. Then the next person shares their screen. It's like watching paint dry. A session that should be 30 minutes takes 90 because everything is sequential.

**Sam Chen:** Are you using the commenting feature at all for async feedback?

**Rachel Torres:** We tried. It doesn't work for us. Our review cadence is too fast. By the time someone leaves a comment and the other person reads it, the context has shifted. Hardware timelines are tight — we can't afford a 12-hour feedback loop between Austin and Shenzhen. We need everyone in the same canvas, editing at the same time, seeing each other's cursors. Real-time multiplayer, basically.

**Hannah:** That's really clear feedback. We've been hearing similar signals from other accounts, and real-time collaboration is actively in development. Sam, do you want to speak to where we are?

**Sam Chen:** Yeah, we've been prototyping real-time collaboration internally. It's early, but the architecture is solid — we're using CRDTs for conflict resolution, and we've got basic cursor presence working in our dev environment. We're probably 4 to 6 weeks from something we could show externally.

**Rachel Torres:** That would be incredible. Seriously, that's the feature that would change everything for us. Can we be first in line to test it?

**Hannah:** Absolutely. I'll make sure you're in the beta cohort.

## Serialized Editing Pain (12:45 - 19:00)

**Hannah:** Ben, Rachel mentioned the sequential workflow. Can you talk about what that looks like day to day for you?

**Ben Harlow:** Yeah, so Rachel and I work on the same prototypes a lot. She owns the data visualization layer — charts, graphs, real-time sensor feeds — and I own the control panel components — buttons, sliders, configuration forms. In a perfect world, we'd both be editing the same project at the same time. She's tweaking the dashboard grid layout, I'm adjusting the motor control sliders, and we can see each other's work as it happens.

**Sam Chen:** And today?

**Ben Harlow:** Today it's fully serialized. I work on my part, export it, hand it to Rachel, she works on her part. Then we get on a call to merge everything together. If I could edit a control panel prototype while Rachel reviews the data viz layer, we'd cut our review cycles in half. Right now we're serialized — I wait for her, she waits for me. It's the biggest time sink in my week.

**Hannah:** How much time would you estimate you lose to that serialization?

**Ben Harlow:** Conservatively? Four to five hours a week across the team. And that's just the direct time. The context-switching cost is probably another two to three hours. You finish your work, move on to something else, then get pulled back when Rachel's ready. It breaks flow.

**Rachel Torres:** And multiply that by the number of projects. We have 11 active projects right now. Even if only half of them have multi-person editing needs, it adds up fast.

## Python/FastAPI Support (19:00 - 25:30)

**Hannah:** Makes sense. I want to make sure we cover your other items too. Rachel, you mentioned Python support in your pre-call notes?

**Rachel Torres:** Right. So a lot of our backend infrastructure is Python — FastAPI specifically. When we prototype a frontend in Forge, we still have to hand-roll the backend separately. If Forge could generate FastAPI endpoints alongside the React frontend, we could validate full-stack flows in a single tool. Right now we prototype the UI in Forge and then spend a day wiring up a FastAPI backend manually. It would be amazing to describe an API contract in the prompt and have Forge scaffold the whole thing.

**Sam Chen:** That's interesting. Today we support React, Next.js, and Vue on the frontend side. Python backend support is something we've discussed but haven't scoped yet. What would the ideal experience look like for you?

**Rachel Torres:** I'd want to say something like "build me a dashboard that shows real-time motor telemetry, with a FastAPI backend that reads from our MQTT broker." And get both the frontend and the API scaffold. Even if the backend is just stubs with the right route structure and Pydantic models, that saves us a ton of boilerplate.

**Ben Harlow:** The firmware prototyping angle is also relevant. Our firmware team writes Python scripts for hardware testing. If they could prototype a simple web UI on top of those scripts using Forge, that would pull a whole new user group into the platform.

**Hannah:** Great context. I'll flag this to the framework support workstream. It's not on the immediate roadmap, but hearing the specific use cases helps us prioritize.
