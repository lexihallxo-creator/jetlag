## Real-Time Collaboration Demo (3:00 - 18:30)

**Sam Chen:** Okay, so watch the left window. I'm going to click into a button component and start editing the label text. You can see on the right window — there's my cursor, moving in real time. And when I type, the text updates on both sides within about 200 milliseconds.

**Rachel Torres:** Oh my God.

**Sam Chen:** Now I'll switch to the right window and start editing a different component — let's say this card layout. Watch the left side. You can see the second cursor appear, and now both users are editing different parts of the same project simultaneously.

**Rachel Torres:** I literally just turned to Ben — this is it. This is the thing that makes Forge irreplaceable for us. When I saw the real-time collaboration prototype, I literally turned to my team and said — this is it. We do hardware design reviews with engineers in three time zones. Right now we're screen-sharing and shouting over each other. If I can have four engineers editing the same prototype simultaneously with live cursors? That changes everything about how we do reviews.

**Ben Harlow:** I've been waiting for this since we signed up. This is the feature that turns Forge from a prototyping tool into our daily driver. Seriously, Sam, this is great.

**Hannah:** That's incredible to hear. Sam, can you talk about the technical approach?

**Sam Chen:** Sure. We're using CRDTs — conflict-free replicated data types — under the hood. That means edits from different users can happen simultaneously without a central coordination server making decisions. Each user's edits are applied locally and then synced. In the vast majority of cases, edits merge cleanly because people are working on different parts of the project.

**Lena Zhao:** What happens when they're not? When two people edit the same thing at the same time?

**Sam Chen:** Right now, last-write-wins at the character level — similar to Google Docs. We also have presence indicators in the top right corner, a list of active users with colored dots matching each cursor on the canvas.

**Hannah:** Lena, let's go to your feedback. What are you thinking?

**Rachel Torres:** That would be huge for our reviews. Right now when someone says "I'm looking at the header," we ask "which header?" If the cursor tells me the component, that removes a whole category of confusion.

**Taylor Brooks:** What about a sidebar panel listing all active users and their current component? Like a table of contents for people.

**Lena Zhao:** Yes. Both. Inline cursor labels for quick glance, sidebar for the overview. With four or five people in a project, you want to scan and see where everyone is.

## Conflict Resolution Discussion (28:00 - 36:00)

**Lena Zhao:** Okay, now back to the conflict question. Last-write-wins is fine for text fields, but what about structural edits? If Rachel is rearranging the layout grid and I'm adding a new component to a cell she's about to move, what happens?

**Sam Chen:** In the current implementation, CRDTs handle this at the tree level. Both edits would be preserved — Rachel's layout change and your new component — but the final position might not be what either of you intended. We don't have a UX for surfacing that kind of conflict yet.

**Lena Zhao:** That's the gap I'm worried about. I don't want a hard lock — but I want to know. A subtle highlight on components someone else is actively modifying. That way I can decide: wait two seconds for Rachel to finish, or go ahead.

**Rachel Torres:** Exactly. Awareness is more important than prevention. We trust each other — we just need to see what's happening.

**Taylor Brooks:** That aligns with our direction. We've been calling it "soft presence" internally — you see where people are and what they're touching, but nothing stops you.

**Sam Chen:** Implementation-wise, we emit events on component selection or editing. The UI subscribes and renders the highlight. Lightweight.

**Lena Zhao:** And one more nuance — when a CRDT merge produces something unexpected, can you show an inline diff? "Merged edit from Rachel" with the option to undo. I don't want silent merges where the output is wrong and nobody noticed.

**Sam Chen:** Haven't built that yet, but great idea. I'll add it to the spec.

## Design Token Sync (36:00 - 42:00)

**Lena Zhao:** One more thing, totally separate topic. We maintain a design system in Figma — colors, typography, spacing, the whole kit. Every time I start a new Forge project, I have to re-specify all of that in the prompt. It's tedious and error-prone.

**Taylor Brooks:** You want Forge to pull those tokens from Figma automatically?

**Lena Zhao:** Exactly. If I could connect my Figma file and have Forge import the design tokens as project-level configuration, every generation would use our brand system by default. That would save me 10 minutes per project on the initial prompt, and it would eliminate the drift we get when someone forgets to specify the right shade of blue.

**Taylor Brooks:** How do you maintain your tokens today — Figma variables, or a plugin like Tokens Studio?

**Lena Zhao:** Figma variables. Published library with all our primitives. Our devs reference them in code as CSS custom properties.

**Taylor Brooks:** So the ideal flow: connect a Figma file, Forge reads the variables, maps them to CSS custom properties or a Tailwind config, and every generation uses those values. I'll dig into the Figma Variables API and draft a proposal.
