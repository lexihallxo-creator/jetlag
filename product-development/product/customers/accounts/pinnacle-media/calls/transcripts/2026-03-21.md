## Figma Import and Design-to-Code (4:30 - 18:45)

**Hannah:** Leo, last time we talked you mentioned you'd been looking at other tools for design-to-code. Can you give us an update on where that stands?

**Leo Martinez:** I'm going to be straight with you — I've been evaluating Lovable for the past three weeks because they're further along on design-to-code. But their generated code is garbage compared to Forge. I want to stay. Give me a reason to.

**Hannah:** I hear you. And I respect the directness. Sam, can you talk about where we are on the design-to-code work?

**Sam Chen:** Yeah. So we've been doing research into Figma API integration for the past two months. The challenge is doing it well versus doing it fast. A lot of the competitors that have Figma import generate code that structurally matches the design but uses absolute positioning, hardcoded pixel values, and doesn't produce responsive layouts. That's not the bar we want to hit.

**Leo:** That's exactly what I'm seeing with Lovable. The output looks right on one screen size and falls apart everywhere else. But Sofia, do you want to explain why this matters so much to us?

**Sofia Huang:** I spend 40 hours a week in Figma. My designs are pixel-perfect. And then when we go to build, an engineer has to manually translate everything. If I could drag a Figma frame into Forge and get code that matches my design, that would eliminate the biggest bottleneck on our team.

**Taylor Brooks:** Sofia, can I ask — when you say pixel-perfect, are you using auto-layout in Figma consistently? And are you working with design tokens or a component library?

**Sofia:** Yes to both. We have a full component library in Figma with design tokens for colors, spacing, typography. Everything uses auto-layout. I've been very disciplined about it because I knew eventually we'd want to bridge to code.

**Taylor:** That's actually great news, because well-structured Figma files are dramatically easier to convert to good code. The projects where Figma import breaks down are the ones with freeform, absolute-positioned designs.

**Sam:** Right. And Sofia, the fact that you're using auto-layout and tokens means when we do build this, your files would be ideal candidates. We're targeting a semantic import — meaning we'd parse your auto-layout as flexbox, your tokens as CSS custom properties, and your components as React components.

**Sofia:** That sounds incredible. What's the timeline?

**Hannah:** I don't want to give you a date I can't commit to. What I can tell you is it's our highest-priority feature on the Generation Quality roadmap. I'll follow up by end of week with a more concrete timeline once I sync with the team.

**Leo:** Hannah, I need something more concrete than "highest priority." My renewal is in May. If I can't tell my CEO that this is coming, I have to make a different call.

**Hannah:** Understood. I'll get you a real answer by Tuesday.

## Performance and Large Datasets (18:45 - 29:00)

**Hannah:** Let's talk about the content catalog performance issues you flagged in your support ticket last week.

**Leo Martinez:** So we built a content management tool in Forge for our media library. Articles, videos, images, metadata — the whole thing. With our test data, maybe 500 items, it's beautiful. Fast, responsive, great UI. But our production catalog has 12,000 items and the app becomes unusable. We need virtual scrolling and server-side filtering. The generated code does client-side filtering by default, which obviously doesn't scale.

**Sam Chen:** That's a fair criticism. Right now the generated code loads all data into memory and filters client-side. For the dataset sizes most of our users work with, that's fine. But 12K items is a different beast. There are two things we can do. Short-term, I can help you refactor the existing generated code to add pagination and server-side filtering. Longer-term, we need the generation model to be smarter about detecting when datasets are large and automatically choosing the right patterns.

**Leo:** The short-term fix would help. But honestly, I shouldn't have to refactor generated code. The whole point of Forge is that it generates production-quality output. If it can't handle a realistic dataset size, that's a product gap.

**Sam:** Agreed. I'm going to look into adding virtual scrolling as a default pattern when the data source exceeds a configurable threshold. It's the kind of thing the model should handle automatically.

**Hannah:** Leo, would you be willing to share the content catalog project with Sam so he can use it as a test case for the performance work?

**Leo:** Yeah, I can do that. I'll share the project tomorrow.

**Sofia Huang:** While we're on performance — the image-heavy pages are also slow. We have gallery layouts with 200+ high-res images and there's no lazy loading. Every image loads on page render.

**Sam:** That's another one where we should be generating better defaults. Lazy loading with intersection observer should be standard for image grids. I'll flag that alongside the virtual scrolling work.

## Template Flexibility and Media Layouts (29:00 - 37:30)

**Taylor Brooks:** Sofia, you mentioned earlier that the template customizer doesn't work well for your media-specific needs. Can you walk me through what you're trying to build?

**Sofia Huang:** Sure. We need three layout patterns constantly. First, video grids — think Netflix-style rows with hover previews and metadata overlays. The existing grid templates are too basic; they don't handle aspect ratio management for mixed video formats. Second, image galleries with lightbox — a masonry layout that opens into a full-screen lightbox with swipe navigation. Third, media detail pages with rich metadata sidebars, related content carousels, and embedded players.

**Taylor:** Those are all patterns we don't have good template coverage for. Our templates were originally built around SaaS dashboards and landing pages. Media is a vertical we haven't invested in yet.

**Sofia:** And that's frustrating because the foundation is there. What I love about Forge is that the UIs actually look good. Other AI tools generate things that look like a developer designed them. Forge gives me layouts with proper spacing, typography hierarchy, and responsive breakpoints. If you add Figma import on top of that quality, it's game over for everyone else.

**Taylor:** That's really validating to hear. Can we schedule a separate session where you walk me through your top 5 layout patterns? I want to understand exactly what you need so I can prioritize the right templates.

**Sofia:** I'd love that. I can share my Figma files ahead of time so you can see what we're working with.

**Taylor:** Perfect. I'll send a calendar invite for next week.

**Leo Martinez:** One more thing on templates — custom fonts. We have brand fonts that we license. There's no way to load them into Forge projects right now. We end up with generated UIs that look great structurally but use the wrong typography, and then we have to manually swap fonts across every file.

**Sam Chen:** That's a smaller lift than some of the other things we've discussed. I think we could support custom font uploads in the project settings relatively quickly. Let me check with the team.
