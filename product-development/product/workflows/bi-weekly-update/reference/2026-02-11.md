# February 2026: What the Team Is Building Now

Forge has three objectives this quarter: (1) raise generation quality to production-grade for complex apps, (2) build the enterprise features needed to convert pilot customers, and (3) expand platform integrations so generated apps fit into customers' existing stacks.

## Generation Quality (OKR 1)

**Goal:** 80% of generated apps pass production readiness review without manual code changes

Enterprise pilots have surfaced that single-page generation is table stakes. Meridian and CloudKitchen both need multi-page apps with shared state, auth flows, and responsive layouts. This OKR targets the gap between "impressive demo" and "actually deployable."

| Workstream | Why It Matters | Status |
|---|---|---|
| **Multi-Page App Generation** | Pilots need full applications, not isolated pages | Engineering: core routing + layout system in review. Target: Feb 28 |
| **Component Library Support** | Enterprise teams bring their own design systems | Design complete. Engineering starting sprint 7 |
| **State Management Patterns** | Complex apps need predictable state handling | Research phase. Evaluating Zustand vs. Jotai patterns for generated code |
| **Design Token Integration** | Customers need brand consistency across generated apps | Spec complete. Blocked on component library work |
| **Responsive Layout Engine** | 40% of pilot users access generated apps on mobile | Shipped v1 in Jan. Iterating on edge cases from Meridian feedback |

## Enterprise & Deployment (OKR 2)

**Goal:** Close 3 enterprise pilots with $50K+ ARR each

Every enterprise prospect asks about SSO, audit logging, and team workspaces in the first call. These are hard blockers for procurement approval.

| Workstream | Why It Matters | Status |
|---|---|---|
| **SSO / SAML Integration** | Hard requirement for Meridian (healthcare compliance) | Engineering: 60% complete. SAML flow working, SCIM provisioning next |
| **Audit Logging** | Nexus Financial requires full audit trail for compliance | Spec complete. Engineering starting sprint 8 |
| **Team Workspaces** | Collaboration is the #1 request from teams > 5 people | Design in progress. User research with CloudKitchen scheduled for Feb 14 |
| **CI/CD Pipeline** | Customers want to deploy generated apps through their existing pipelines | Shipped GitHub Actions integration. GitLab CI support in sprint 7 |
| **Environment Management** | Enterprise teams need dev/staging/prod separation | Early design phase. Gathering requirements from pilots |

## Platform & Integrations (OKR 3)

**Goal:** Support the top 3 database providers and top 2 auth providers natively

Generated apps that require manual integration setup create friction that kills conversion. If Forge generates a full-stack app, it needs to generate the full stack.

| Workstream | Why It Matters | Status |
|---|---|---|
| **Git Provider Integration** | Every team needs version control for generated code | GitHub shipped. GitLab in sprint 7. Bitbucket deprioritized |
| **Database Connectors** | Full-stack generation requires database setup | PostgreSQL shipped. Supabase connector in review. MongoDB next |
| **Auth Provider Support** | Auth is the first integration every app needs | Auth0 shipped. Firebase Auth in development. Clerk integration planned |
| **Plugin / Extension API** | Power users want to extend generation with custom logic | Early research. Gathering requirements from Aether Logistics |
| **Cloud Provider Export** | Enterprise teams deploy to their own infrastructure | AWS CDK export in beta. GCP and Azure scoped for Q2 |

## Designs Complete (Ready for Engineering)

- Team workspace invite flow (pending component library)
- Audit log viewer and search UI
- Multi-page app navigation patterns (tabs, sidebar, breadcrumbs)

## What We're Hearing

**Enterprise buyers are converting on quality, blocking on compliance.** The generation quality improvements shipped in January are landing well with enterprise prospects. Meridian expanded their pilot from 3 to 8 internal tools after seeing the responsive layout improvements, and Nexus Financial's engineering lead called the code output "surprisingly production-ready." But compliance features (SSO, audit logging) remain hard blockers for procurement.

- *Compliance as gatekeeper:* Meridian's CISO requires SAML SSO before any tool touches patient-adjacent data. Nexus Financial's compliance team needs full audit trails. These aren't nice-to-haves.
- *Collaboration as accelerator:* CloudKitchen and Aether both highlighted that their teams generate apps in isolation and then struggle to share and iterate. Team workspaces could be the unlock for multi-seat expansion.

**Free-tier users are our best competitive intelligence source.** TrueNorth Analytics switched from v0 because v0 generates components but not full apps. Their feedback on AWS deployment friction directly validates our cloud export roadmap.

- *Deployment is the new battleground:* v0 and Lovable both stop at code generation. Our CI/CD pipeline and cloud export work is a meaningful differentiator.

## Highlights

1. **Meridian Health expanding pilot 2.5x.** Their team went from 3 to 8 internal tools on Forge, driven by responsive layout improvements. SSO remains the single blocker for enterprise-wide rollout.
2. **Nexus Financial compliance team is the buyer.** Engineering loves the product but procurement won't approve without audit logging. Sprint 8 start aligns with their March evaluation deadline.
3. **CloudKitchen: collaboration is the unlock for multi-seat.** Their 8-person team currently generates apps individually and shares via Slack. Team workspaces could convert them from 2 seats to 8.
4. **v0 defection validates full-stack strategy.** TrueNorth Analytics switched to Forge specifically because v0 is frontend-only. First concrete evidence that our full-stack approach wins in head-to-head.
5. **AWS export in beta is drawing enterprise interest.** Aether Logistics and TrueNorth both independently asked about deploying to their own infrastructure. Cloud export is higher priority than we estimated.

## Emerging Product Direction

The enterprise buyer ICP is becoming clearer: mid-size engineering teams (8-20 people) at companies where internal tools are a tax on product engineering time. These teams don't want to hire internal tools specialists. They want to generate internal tools from descriptions and iterate quickly.

The conversion path is: eng lead finds Forge -> solo pilot (1-2 seats) -> proves ROI on 3-5 tools -> pushes for team adoption -> compliance/management approves -> full team rollout (8-20 seats).

# Meridian Health: Pilot Exec Summary

**Company:** Healthcare SaaS platform, 200 employees, Series C
**Team:** 12-person engineering team, 4 currently on Forge
**Pilot started:** December 2025
**Pilot scope:** Internal tools for clinical operations team
**Status:** Expanding from 3 to 8 tools. SSO is the blocker for enterprise-wide rollout.

## Pain Points

1. **Internal tools are a tax on product engineering.** Meridian's eng team spends ~30% of time building and maintaining internal tools for clinical operations. Every tool is hand-coded React + Node, and they have 15+ internal apps.
2. **Brand and UX inconsistency.** Each internal tool was built by a different engineer with different patterns. Clinical staff complain about the learning curve.
3. **Compliance gates slow everything down.** Any tool touching patient-adjacent data requires CISO approval. The approval process takes 4-6 weeks per tool.

## Forge Use Cases Today

- **Clinical scheduling dashboard:** Generated full CRUD app for managing provider schedules. Replaced a 3-month internal project.
- **Patient feedback viewer:** Generated a read-only dashboard pulling from their survey API. Operations team uses it daily.
- **Inventory tracker:** Supply chain team requested an internal tool. Generated in 2 hours, previously estimated at 2 sprints.

## Feature Gaps

| Gap | Impact | Quote |
|---|---|---|
| SSO / SAML | Blocker for enterprise rollout | *"My CISO won't approve anything without SAML. Period."* - VP of Engineering |
| Component library support | Inconsistent UI across generated tools | *"We need these to look like Meridian tools, not generic apps."* - Lead Engineer |
| On-prem export | HIPAA-adjacent compliance | *"Patient data can never touch a third-party server."* - CISO |

# CloudKitchen: Pilot Exec Summary

**Company:** Restaurant operations platform, 80 employees, Series B
**Team:** 8-person engineering team, 2 currently on Forge
**Pilot started:** January 2026
**Pilot scope:** Internal dashboards for restaurant operations data
**Status:** Strong individual usage. Team workspaces needed for broader adoption.

## Pain Points

1. **Dashboard sprawl.** CloudKitchen has 20+ internal dashboards built with different tools (Retool, custom React, spreadsheets). No consistency, hard to maintain.
2. **Collaboration friction.** Engineers generate apps individually and share screenshots in Slack. No way to hand off, review, or iterate together.
3. **Real-time data requirements.** Restaurant ops requires live data (order volume, kitchen times, driver ETAs). Most prototyping tools can't handle real-time connections.

## Forge Use Cases Today

- **Kitchen performance dashboard:** Real-time view of order completion times by restaurant. Generated with WebSocket connection to their API.
- **Menu pricing analyzer:** Comparative pricing tool pulling from multiple delivery platforms. Product team uses it for pricing decisions.

## Feature Gaps

| Gap | Impact | Quote |
|---|---|---|
| Team workspaces | Blocker for team adoption | *"I love this tool but I can't share what I build with my team."* - Senior Engineer |
| Custom data connectors | Need proprietary API support | *"We need to plug in our restaurant management API, not just generic REST."* - CTO |
| Mobile-first templates | Ops managers use tablets in kitchens | *"Half our users are on iPads in restaurant kitchens."* - VP of Operations |
