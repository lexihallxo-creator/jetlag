## Summary

This RFC defines the data pipeline for ingesting, transforming, and warehousing search events generated by the Project Search (Cmd+K) feature. The pipeline moves search interaction events from the Forge frontend through Segment to Snowflake, where they are modeled into a fact table (`fact_search_events`) and a dimension table (`dim_search_results`) for analytics consumption.

## Motivation

The Project Search feature generates two categories of data that the analytics team needs:

1. **Search behavior data** -- What users search for, how often, and whether they find what they need. This drives the search usage rate, zero-results rate, and click-through rate metrics defined in the PRD.

2. **Search quality data** -- Which results are shown, which are clicked, and at what position. This enables relevance tuning and ranking improvements.

Today, Forge's analytics pipeline handles generation events and deployment events but has no infrastructure for search-specific event types. This RFC extends the existing Segment-to-Snowflake pipeline to cover search events with appropriate schema design, quality checks, and backfill support.

## Proposed Schema Design

### `fact_search_events`

The primary fact table. One row per search query execution.

```sql
CREATE TABLE analytics.forge.fact_search_events (
    event_id                VARCHAR(36)     NOT NULL,
    user_id                 VARCHAR(36)     NOT NULL,
    org_id                  VARCHAR(36),
    session_id              VARCHAR(64)     NOT NULL,
    query_text              TEXT            NOT NULL,
    query_length            INTEGER         NOT NULL,
    result_count            INTEGER         NOT NULL DEFAULT 0,
    result_count_projects   INTEGER         NOT NULL DEFAULT 0,
    result_count_templates  INTEGER         NOT NULL DEFAULT 0,
    result_count_actions    INTEGER         NOT NULL DEFAULT 0,
    clicked_result_id       VARCHAR(36),
    clicked_result_type     VARCHAR(30),
    clicked_position        INTEGER,
    filters_applied         VARIANT,
    time_to_click_ms        INTEGER,
    is_zero_results         BOOLEAN         NOT NULL,
    search_latency_ms       INTEGER         NOT NULL,
    subscription_tier       VARCHAR(20)     NOT NULL,
    platform                VARCHAR(20)     NOT NULL,
    created_at              TIMESTAMP_NTZ   NOT NULL,
    loaded_at               TIMESTAMP_NTZ   NOT NULL DEFAULT CURRENT_TIMESTAMP(),

    CONSTRAINT pk_fact_search_events PRIMARY KEY (event_id)
);

-- Clustering for common query patterns
ALTER TABLE analytics.forge.fact_search_events
    CLUSTER BY (created_at, user_id);
```

### `dim_search_results`

Dimension table capturing the result set returned for each search. One row per result per search event. Enables analysis of result quality and position bias.

```sql
CREATE TABLE analytics.forge.dim_search_results (
    result_id               VARCHAR(36)     NOT NULL,
    event_id                VARCHAR(36)     NOT NULL,
    content_id              VARCHAR(36)     NOT NULL,
    content_type            VARCHAR(30)     NOT NULL,
    position                INTEGER         NOT NULL,
    title                   TEXT            NOT NULL,
    relevance_score         FLOAT,
    was_clicked             BOOLEAN         NOT NULL DEFAULT FALSE,
    created_at              TIMESTAMP_NTZ   NOT NULL,
    loaded_at               TIMESTAMP_NTZ   NOT NULL DEFAULT CURRENT_TIMESTAMP(),

    CONSTRAINT pk_dim_search_results PRIMARY KEY (result_id),
    CONSTRAINT fk_dim_search_results_event FOREIGN KEY (event_id)
        REFERENCES analytics.forge.fact_search_events (event_id)
);

ALTER TABLE analytics.forge.dim_search_results
    CLUSTER BY (created_at, event_id);
```

## Data Pipeline

### Architecture

```
Forge Frontend
    │
    ▼
Segment (track call: "Search Executed", "Search Result Clicked")
    │
    ▼
S3 (segment-forge-events bucket, partitioned by date)
    │
    ▼
Snowpipe (auto-ingest from S3 notifications)
    │
    ▼
analytics.forge.raw_search_events (staging)
    │
    ▼
dbt models (transform + dedupe + enrich)
    │
    ├──▶ analytics.forge.fact_search_events
    └──▶ analytics.forge.dim_search_results
```

### Step 1: Segment Event Tracking

Two Segment track events are emitted from the frontend:

**Event: `Search Executed`**

Fired when a search query returns results (including zero results). Properties:

| Property | Type | Description |
|----------|------|-------------|
| `event_id` | string | UUID generated client-side |
| `query_text` | string | Search query string |
| `query_length` | integer | Character count |
| `result_count` | integer | Total results returned |
| `result_count_projects` | integer | Project results |
| `result_count_templates` | integer | Template results |
| `result_count_actions` | integer | Action results |
| `filters_applied` | object | Active filters |
| `search_latency_ms` | integer | Server response time |
| `is_zero_results` | boolean | Whether zero results returned |

**Event: `Search Result Clicked`**

Fired when a user clicks a search result. Properties:

| Property | Type | Description |
|----------|------|-------------|
| `event_id` | string | References the parent search event |
| `clicked_result_id` | string | ID of the clicked result |
| `clicked_result_type` | string | Content type |
| `clicked_position` | integer | 1-indexed position |
| `time_to_click_ms` | integer | Time from search to click |

### Step 2: S3 Landing

Segment delivers events to the `s3://segment-forge-events/search/` prefix. Files are partitioned by date:

```
s3://segment-forge-events/search/dt=2026-03-22/search_executed_001.json.gz
s3://segment-forge-events/search/dt=2026-03-22/search_result_clicked_001.json.gz
```

### Step 3: Snowpipe Ingestion

A Snowpipe auto-ingest pipe monitors the S3 prefix via SQS notifications:

```sql
CREATE PIPE analytics.forge.pipe_search_events
    AUTO_INGEST = TRUE
AS
COPY INTO analytics.forge.raw_search_events
FROM @analytics.forge.segment_s3_stage/search/
FILE_FORMAT = (TYPE = JSON, STRIP_OUTER_ARRAY = TRUE)
MATCH_BY_COLUMN_NAME = CASE_INSENSITIVE;
```

Latency target: events available in the raw table within 2 minutes of S3 delivery.

### Step 4: dbt Transformation

Two dbt models transform raw events into the fact and dimension tables:

**`stg_search_events`** (staging model):
- Deduplicates by `event_id` (keeps earliest `loaded_at`).
- Joins `Search Executed` and `Search Result Clicked` events on `event_id`.
- Enriches with `subscription_tier` from the `users` dimension.
- Casts types and applies naming conventions.

**`fact_search_events`** (mart model):
- Incremental model (`merge` strategy on `event_id`).
- Adds `is_zero_results` boolean derivation.
- Applies data quality assertions (non-null `user_id`, valid `subscription_tier`).

**`dim_search_results`** (mart model):
- One row per result per search event.
- Derived from the `search_results` array in the raw `Search Executed` event payload.
- Flags `was_clicked = TRUE` when the result's `content_id` matches the `clicked_result_id` from the corresponding click event.

dbt models run on a 15-minute incremental schedule via the existing dbt Cloud job.

## Key Queries

**Daily search usage rate (DAU-based):**

```sql
SELECT
    DATE_TRUNC('day', fse.created_at) AS dt,
    COUNT(DISTINCT fse.user_id) AS searchers,
    dau.user_count AS dau,
    ROUND(searchers * 100.0 / NULLIF(dau.user_count, 0), 2) AS usage_rate_pct
FROM analytics.forge.fact_search_events fse
JOIN analytics.forge.daily_active_users dau
    ON DATE_TRUNC('day', fse.created_at) = dau.event_date
WHERE fse.created_at >= DATEADD('day', -30, CURRENT_DATE())
GROUP BY 1, 3
ORDER BY 1 DESC;
```

**Zero-results analysis with top queries:**

```sql
SELECT
    LOWER(TRIM(query_text)) AS normalized_query,
    COUNT(*) AS occurrences,
    COUNT(DISTINCT user_id) AS unique_users
FROM analytics.forge.fact_search_events
WHERE is_zero_results = TRUE
  AND created_at >= DATEADD('day', -7, CURRENT_DATE())
GROUP BY 1
ORDER BY occurrences DESC
LIMIT 25;
```

**Click position distribution:**

```sql
SELECT
    clicked_position,
    COUNT(*) AS clicks,
    ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER (), 2) AS pct
FROM analytics.forge.fact_search_events
WHERE clicked_result_id IS NOT NULL
  AND created_at >= DATEADD('day', -30, CURRENT_DATE())
GROUP BY 1
ORDER BY 1;
```

## Data Quality & Validation

| Check | Type | Threshold | Action |
|-------|------|-----------|--------|
| `event_id` uniqueness | dbt test | 0 duplicates | Fail pipeline run |
| `user_id` not null | dbt test | 0 nulls | Fail pipeline run |
| `result_count` >= 0 | dbt test | 0 violations | Fail pipeline run |
| `search_latency_ms` > 0 | dbt test | < 1% violations | Warn |
| Row count variance | custom macro | < 50% day-over-day drop | Alert to #forge-data-alerts |
| Freshness | dbt source freshness | < 30 minutes | Alert to #forge-data-alerts |
| `subscription_tier` valid enum | dbt accepted_values | `free`, `pro`, `teams`, `enterprise` | Fail pipeline run |
| `clicked_position` within range | dbt test | 1-25 when not null | Warn |

Alerts are routed to the `#forge-data-alerts` Slack channel. Pipeline failures block downstream dashboard refreshes until resolved.

## Migration & Backfill Plan

### New tables (no migration required)

Both `fact_search_events` and `dim_search_results` are new tables. No existing data needs to be migrated.

### Backfill strategy

1. **Pre-launch:** No backfill needed. Tables are created empty before the search feature goes live.
2. **Beta period backfill:** During the beta rollout (10% of users), events are tracked from day one via Segment. The Snowpipe will ingest these in real-time. If any events are missed during initial pipeline setup, a manual backfill can be run from the S3 archive:

```sql
COPY INTO analytics.forge.raw_search_events
FROM @analytics.forge.segment_s3_stage/search/
PATTERN = 'dt=2026-03-.*/.*\.json\.gz'
FILE_FORMAT = (TYPE = JSON, STRIP_OUTER_ARRAY = TRUE)
MATCH_BY_COLUMN_NAME = CASE_INSENSITIVE;
```

3. **GA launch:** By the time search reaches GA, the pipeline will have been running for 2-3 weeks during beta. No additional backfill is required for GA.

### Rollback

If the search feature is rolled back via feature flag, the pipeline continues to run but receives no new events. Existing data is retained for post-mortem analysis. To fully decommission:

1. Disable the Snowpipe: `ALTER PIPE analytics.forge.pipe_search_events SET PIPE_EXECUTION_PAUSED = TRUE;`
2. Disable the dbt models by removing them from the job selector.
3. Tables are retained for 90 days, then dropped if the feature is permanently cancelled.
